结合现有的组件和数据模型，以下是为您规划的**收款管理 (Payment Management)** 页面 UI 设计建议。整体风格可以保持与现有“订单管理”一致的现代化 B2B 后台风格。

### 1. 顶部操作与筛选区 (Header & Filters)

* **标题与主操作：** 左侧显示页面标题“收款管理”，右侧放置一个主色调的 `Button`（例如：**+ 新增收款**）。
* **快速筛选区：**
* **搜索框 (Input)：** 支持模糊搜索：按“收款编号” (paymentNo)， 按“客户名称” (customerName)，按时间段筛选。
* **下拉筛选 (Select)：** * 按“收款账户” (bankAccount) 筛选。
* 按“分配状态”筛选（通过计算 `unallocatedBalance` 判断是“未分配”、“部分分配”还是“已分配完毕”）。
* 按“款项类型” (paymentType) 筛选。

* **日期选择器：** 按“到账日期” (receivedDate) 筛选区间。



### 2. 核心数据表格 (Data Table)

使用现有的 `@tanstack/react-table` 构建核心列表。建议展示以下列，以保证信息密度和可读性：

* **收款编号 & 到账时间：** 将 `paymentNo` 作为主标识加粗显示，下方以小号灰色字体辅以 `receivedDate`，节省横向空间。
* **客户名称：** `customerName`。
* **实收金额：** `receivedAmount`，建议右对齐，并带上货币符号。
* **待分配余额：** `unallocatedBalance`。为了突出重点，当余额大于 **0** 时，可以使用醒目的颜色（如红色或橙色字体）提示用户还有款项未核销。
* **收款账户/方式：** 将 `bankAccount` 和 `paymentMethod` 组合展示。
* **分配状态 (Badge)：** 这是一个衍生列，使用 `Badge` 组件增强视觉效果：
* 绿色 Badge：**已全额分配** (`unallocatedBalance` == 0)
* 黄色 Badge：**部分分配** (`unallocatedBalance` < `receivedAmount` 且 > 0)
* 红色/灰色 Badge：**未分配** (`unallocatedBalance` == `receivedAmount`)


* **操作列：** 提供 `Button` (变体为 ghost 或 link) 用于“查看详情”和“核销订单”。

### 3. 收款详情页 / 侧边抽屉 (Detail View)

当用户点击某一条收款记录的“查看详情”时，建议使用抽屉 (Drawer) 或 `Dialog` 弹窗展示，避免页面跳转带来的割裂感。内部可以使用 `Tabs` 组件分为两个主要模块：

* **Tab 1：基本信息 (Card 布局)**
* 使用网格布局展示收款的完整只读字段（包含 `remarks` 备注等）。
* 顶部可以放置一个 `Progress` 进度条，直观展示该笔收款的**核销进度**（计算公式：`(实收金额 - 待分配余额) / 实收金额`）。


* **Tab 2：关联核销/订单记录 (Table)**
* 展示一对多关系。列出这笔收款具体抵扣了哪些订单、每次抵扣的金额和日期。



### 4. 交互提示 (Toast)

* 在用户执行“新增收款”、“更新收款”或“完成一次订单核销”时，调用 `sonner` 弹出成功或失败的 Toast 提示，提供即时反馈。

### 3. 技术实现上的现实考量

这是一个非常聪明且务实的过渡方案。既然你已经在 Airtable 中配置好了 AI 字段（如截图所示的“实收金额（自动提取）”），完全可以通过 Airtable API 将其作为你的“后端 AI 引擎”来使用。

这种方案的核心逻辑是：**利用 Airtable 的异步处理能力，前端只负责触发和展示。**

---

### 实现方案建议

由于 Airtable AI 字段在填入原文后需要几秒钟的“计算时间”（即你截图中看到的“Baked for 33s”过程），你的 UI 设计需要考虑这个延迟：

#### 1. 交互流程设计

1. **提交原文：** 用户在 Web 界面粘贴“银行收款通知”，点击“识别”后，程序通过 API 将该文本写入 Airtable 的相应记录。
2. **等待状态：** 前端 UI 进入 Loading 状态（例如：实收金额输入框显示一个转圈的动画）。
3. **轮询获取 (Polling)：** 前端每隔 1-2 秒请求一次 Airtable API，检查该记录的 AI 字段是否有值。
4. **自动填充：** 一旦 Airtable 计算完成返回数值，前端立即更新 `receivedAmount` 等字段。

#### 2. UI 界面调整建议

为了配合这个逻辑，UI 可以做如下微调：

| 阶段 | UI 表现 | 背后动作 |
| --- | --- | --- |
| **输入阶段** | 文本框下方显示“点击识别”按钮。 | 准备调用 API。 |
| **识别中** | 按钮变为“AI 正在解析中...”，实收金额框显示置灰占位符。 | 调用 Airtable `PATCH` 接口更新文本。 |
| **完成识别** | 金额框自动填入数值，并伴随一个高亮闪烁效果。 | 轮询监听到字段变化，获取 `fields["实收金额"]`。 |

### 技术实现要点

* **API 调用：** 使用你熟悉的 React/TypeScript 环境，通过 `fetch` 或 `axios` 调用 Airtable REST API。
* **状态同步：** 因为你已经在做“Airtable 同步功能”，可以复用现有的 API 鉴权和基础类。
* **延迟处理：** Airtable AI 的响应速度取决于内容复杂度。建议在前端设置一个“手动确认”环节，即 AI 填入数值后，用户点击一下才最终生效，防止 AI 偶发性的错误直接进入数据库。

### 潜在限制

* **API 配额：** Airtable 的 API 有每秒 5 次请求的限制。如果你的并发量大，频繁轮询可能会触发限制。
* **时效性：** 这种方案会有 3-10 秒的延迟，虽然不如直接接入 OpenAI API 快，但对于财务登记这种对实时性要求没那么极端的场景完全够用。

* **API 请求耗时：** * 通常单次 Airtable API 请求的往返时间（Latency）在 **$200ms$ 到 $1s$** 之间，具体取决于您的网络环境。
* **API 配额限制：** * Airtable 标准 API 有每秒 **$5$ 次** 请求的硬性限制，如果多个用户同时操作，频繁轮询很容易触发 `429 Too Many Requests` 错误。

---

### 2. 建议的轮询获取策略

为了平衡“反馈速度”和“API 消耗”，我不建议从第 $1$ 秒就开始轮询。建议采用 **“延迟触发 + 固定频率”** 的策略：

| 阶段 | 动作 | 时间点 | 目的 |
| --- | --- | --- | --- |
| **初始等待** | 静默期 | $0s$ - $20s$ | 避开 AI 必经的计算耗时，节省约 $4$-$5$ 次 API 调用。 |
| **首次检查** | 第一次 API 调用 | **第 $20s$** | 尝试获取结果。部分简单文本可能处理较快。 |
| **循环轮询** | 每 **$5s$** 调用一次 | $25s$, $30s$, $35s$... | 保持较低的请求频率，避免触发 API 限流。 |
| **超时停止** | 停止轮询 | **第 $60s$** | 如果一分钟还没结果，可能解析失败，提示用户手动输入。 |

---

### 3. UI 交互上的小建议

既然用户需要等待约 $10$ 秒，我们可以通过 UI 缓解焦虑：

* **进度进度提示：** 在用户点击“识别”后，展示一个带有倒计时的进度条，初始设为 $30$ 秒。这能给用户一个明确的心理预期。
* **非阻塞操作：** 允许用户在 AI 识别期间先填写其他字段（如“客户名称”或“备注”），识别出的金额在后台静默更新。
* **手动刷新按钮：** 轮询期间提供一个小的刷新图标，如果用户觉得已经等够了，可以手动触发一次请求。
