针对你明天的计划，这里有几个基于 Airtable API 规范和项目架构的深度建议，供你参考：

### 1. 设计 Supabase 缓存表结构

* **字段精简**：在创建 Supabase 表（如 `st_orders` 和 `st_payments`）时，务必只包含前端 GUI 必需的字段。根据 Airtable 文档，API 返回的字段会很多，但在 Supabase 中保持轻量化可以提高查询效率。
* **记录 Record ID**：确保每一个缓存表都有一个 `airtable_record_id` 字段。这是未来执行“实时写入”操作时，精确定位 Airtable 记录的唯一凭证。

### 2. 实现 Airtable → Supabase 同步逻辑

* **频率限制预警**：在编写同步脚本时，请务必处理 Airtable 每秒 5 次请求的频率限制。如果数据量较大，建议使用分批次（Batch）请求，并加入适当的延迟（如 200ms），以避免触发 `429 Too Many Requests` 错误。
* **利用 View 过滤**：既然你已经设计了 View 过滤方案，在调用 API 时使用 `view` 参数。这样你可以直接在 Airtable 界面上控制哪些数据需要同步（例如“仅同步 2026 年活跃订单”），而无需修改同步代码。

### 3. 开发收款核销页面

* **权限校验**：在开发 GUI 提交核销动作时，确保使用的 Token 拥有 `data.records:write` 权限，这是在 Airtable 中创建分配记录（中间表）的必要条件。
* **交互反馈**：由于核销涉及实时写入 Airtable，建议在界面上增加明显的 Loading 状态和错误捕获（如捕获 `422 Invalid Request`），以应对由于网络或权限配置导致的写入失败。



这两个问题直击系统设计的核心：一个是**数据源的过滤规则**，另一个是**系统的可维护性架构**。

我们分两部分来详细拆解，确保德皓 (Dehao WebPage) 的架构既稳健又灵活。

### 一、 关于 `view` 参数在 Airtable 中的详细设置逻辑

在 Airtable API 中，`view` 参数的作用是**“守门员”和“排序员”**。当你向 API 传入一个 `view` 名称或 ID 时，API **只会返回该视图下可见的行（记录）**，并且按照该视图的排序规则返回。

以下是保姆级的设置指导：

#### 步骤 1：在 Airtable 中创建专属的“API 视图”

不要直接使用财务人员日常操作的默认视图（Grid view），因为他们可能会随手修改筛选条件，导致你的 API 突然拉取到错误的数据。

1. 打开 Airtable 的 `PO（订单表）`。
2. 在左侧边栏底部的 "Create..." 区域，点击 **Grid** 新建一个视图。
3. 命名规范：建议命名为 `API_Sync_Active` 或 `API_Pending_Orders`。加上 `API_` 前缀，明确告诉所有协作者：“**这是给程序用的，绝对不要手动改！**”

#### 步骤 2：设置强大的 Filter（过滤规则）

这是 View 参数最核心的价值。在这个新建的视图中，点击顶部的 **Filter**，设置如下条件：

* `Where [出货状况] is not [作废]`
* `And [欠款余额] > 0`
* `And [订单总额] is not empty`
* *(你甚至可以加：`And [创建时间] is within the past 365 days`，防止拉到太老的数据)*

#### 步骤 3：获取 View ID

为了代码的稳定性，建议在 API 中使用 View ID 而不是 View 名称（因为名称如果被修改，API 会报错 `404 Not Found`）。

1. 保持在这个 `API_Sync_Active` 视图下。
2. 观察浏览器的 URL 地址栏，它看起来像这样：
`https://airtable.com/appXXXXXX/tblXXXXXX/viw1234567890ABCD`
3. 复制以 **`viw`** 开头的那段字符串，这就是你的 View ID。

> ⚠️ **高能避坑警告 (开发者常踩的坑)**：
> Airtable 的 `view` 参数**只会过滤行（Rows），不会过滤列（Columns / Fields）**。
> 即使你在 Airtable 的视图里隐藏了 50 个字段，API 拉取时依然会把这 50 个字段的数据全返回给你，这会极大拖慢响应速度。
> **解决办法**：在你的 TypeScript API 请求代码中，除了传入 `view: "viwXXXX"`, 必须同时传入 `fields: ["po_no", "balance", "client_name"]` 参数，明确告诉它只需要这几列。

### 二、 将 Base ID, Table ID, View ID 放入后台 UI 配置？

**结论是：可以做，但确实会显著增加系统的复杂性。**

对于德皓 WebPage 目前的阶段（第一阶段），我们需要权衡投入产出比。以下是深度剖析：

#### 方案对比

| 维度 | 方案 A：写死在环境变量 `.env` (当前计划) | 方案 B：放入前端后台 UI (存入 Supabase 数据库) |
| --- | --- | --- |
| **部署与修改** | 修改 ID 需要更改服务器 `.env` 文件并重启服务。 | 管理员直接在网页上修改，点击保存即生效。 |
| **灵活性** | 较低。只能对接固定的那几个表和视图。 | **极高**。未来可以做成“多租户”或随时切换测试库。 |
| **代码复杂度** | **极低**。读取变量只需 `process.env.BASE_ID`，是同步操作。 | **较高**。每次调用 Airtable 前，都要先异步 `await` 去数据库查一遍最新的 ID，再拼接请求。 |
| **容错率** | 稳定。代码写死的通常不会出错。 | 较脆弱。如果管理员在 UI 里填错了一个字母，整个核销页面瞬间崩溃报错 404。 |

#### 为什么会增加系统复杂性？ (技术阻碍)

如果你选择做 UI 配置，你的 `libs/airtable/client.ts` 逻辑会被重写：
原本它是：`实例化客户端 -> 发送请求`。
现在它必须变成：`读取本地缓存/请求 Supabase 获取最新 ID -> 校验 ID 格式 -> 实例化客户端 -> 发送请求`。
此外，你还要在 Supabase 中新建一张 `st_system_config` 表，并在前端专门写一个“设置表单”页面来管理这些 ID。

#### 💡 给德皓的架构建议 (渐进式演进)：

1. **第一阶段 (MVP) - 坚守环境变量**：
目前你的首要任务是把“收款核销”的业务逻辑跑通。在验证业务价值之前，直接把 Base ID、Table 名称、View ID 写在 `.env` 或 `config/airtable.ts` 里。这能帮你节省至少 1-2 天的开发时间。
2. **第二阶段 (系统升级) - 引入 DB 动态配置**：
当德皓的业务流转顺利，前端 UI 框架也十分稳固了，我们再开一个分支，在 Supabase 建一个配置表，把这些环境变量“迁移”到数据库中，并在后台开放修改权限。

PS: PI号（例如26DH01001）和生产单号（26001单）之间的逻辑关系：
PI号前两位数字表示年份，DH表示公司名称“德皓”，DH后跟两位数字表示进单月份，例如"01"就是1月份进单，最后三位数字表示本年度订单序列号；
而生产单其实就是简化的PI号，把年份和订单序列号取过来组成。
有时候在序列号后面会添加英文字母，这表示客人提供的订单，被我们业务拆分问多个子订单，每个订单号都独立运行。例如26001A和26001B就是按照两个订单来操作。
PI号用于外部和客户沟通使用；生产单号用于内部和生产沟通使用；生产单号也用于和供应商沟通使用。
