## 为什么需要 Function Table？

在简单的业务中，人们习惯直接在“订单”里记录“已收金额”。但在现实业务中：

1. **部分付款**：一个订单分 3 次付清。
2. **合并付款**：客户一次性汇款 10 万，分摊到 5 个不同的订单。
3. **退款/冲抵**：涉及到金额的灵活调整。

**Function Table (Allocation Table)** 作为中间层，记录的是“某笔收款中的多少钱，分配给了哪个订单”。

### 1. Orders (订单表)

存储订单的主体信息，重点在于通过 **Rollup** 字段实时计算未付金额。

| 字段名 (Field Name) | 类型 (Type) | 说明 |
| --- | --- | --- |
| **Order ID** | ID / Formula | 主键，如 "ORD-2026001" |

| **Allocated Amount** | **Rollup** | 从 *Function Table* 汇总已分配的金额 |
| **Balance** | Formula | `Total Amount - Allocated Amount` (待收余额) |

| **Customer** | Linked Record | 关联至客户表 |

## 一、 核心表结构深度优化建议

目前的“中间表”已经承担了核销功能，但为了让财务管理更“深刻”且不容易出错，建议强化以下几个关键字段的校验逻辑：

### 1. 收款\_中间表 (Function Table) - 核心枢纽

这个表不应只存数据，更应起到**校验**作用。

* **本次分配金额 (Amount Applied)**：手动输入，代表从某笔收款中拿多少钱给某个订单。
* **核销类型 (Allocation Type)**：[单选] 建议增加：预付款、尾款、样品费、运费扣除。这对于后续财务对账至关重要。
* **分配比例 (Formula)**：`{本次分配金额} / {订单总金额}`。自动监控这笔款占订单的百分比。

### 2. 收款登记 (Payments) - 资金入口

* **实收金额 (Total Received)**：银行实际到账。
* **已分配总额 (Total Allocated)**：[Rollup] 关联 `收款_中间表` 的“本次分配金额”。
* **待分配余额 (Unallocated Balance)**：[Formula] `{实收金额} - {已分配总额}`。

> **深度建议**：如果这个值 **< 0**，通过 Airtable 条件格式标红。这说明财务分配错了，分出去的钱比收到的还多。
> 
>

### 3. PO (Orders) - 订单终点

* **订单总金额**：订单原始价值。
* **已收总额**：[Rollup] 关联 `收款_中间表` 的“本次分配金额”。
* **未结欠款 (Balance Due)**：[Formula] `{订单总金额} - {已收总额}`。
* **收款状态**：[Formula] 如果 `{未结欠款} <= 0` 则显示 "✅ 已结清"，否则显示 "⏳ 欠款中"。

### 核心法则：中间表到底是什么？

请记住这句“黄金法则”：
**`收款_中间表` 的 1 条记录，永远只代表“一次切蛋糕的动作”。**
它**只能**对应 **1 笔具体的收款** 和 **1 个具体的订单**。

我们用顺展工厂最常见的真实场景来推演，你会瞬间明白为什么需要允许多个关联：

#### 场景：一笔大款，付多个订单（1 对多）

假设危地马拉的客户给你打了一笔 **10,000 美元**（这是 1 笔收款登记）。
这笔钱其实是用来付两个订单的：**PO-A**（欠款 4,000）和 **PO-B**（欠款 6,000）。

在你的系统里，这三张表会这样运作：

1. **收款登记表**

* 只有 **1 条记录**：金额 10,000 美元。
* 这里的 `关联收款_中间表` 字段里，会塞进 **2 个气泡（关联 2 条中间表记录）**。
2. **收款\_中间表（核销表）**

* 这里会产生 **2 条记录**（切了两次蛋糕）：

* **记录 1**：关联这笔 10,000 的收款 + 关联 **PO-A** + 本次分配金额写 **4,000**。
* **记录 2**：关联这笔 10,000 的收款 + 关联 **PO-B** + 本次分配金额写 **6,000**。
3. **PO 表**

* PO-A 里面会看到 1 个中间表关联（进账 4,000）。
* PO-B 里面也会看到 1 个中间表关联（进账 6,000）。

### 总结一下这个框架

把中间表想象成一座**桥梁的桥墩**：

* 桥的左边是“一大笔钱”（收款登记）。

### 四、 边界条件与防错机制 (避坑指南)

在编写 TypeScript 逻辑时，必须提前防御以下问题：

1. **防止超卖 (Double Spending)**：如果两个财务人员同时打开页面处理同一笔收款怎么办？提交到 Supabase 时，后端需要再次验证 Airtable 的最新余额，防止并发超额分配。

### 1. 核心比喻：带着家具的“移动板房”

想象一下你的云服务器（比如腾讯云）是一大片**空地**。

* **传统做法（不盖容器）**：你要在这片空地上建一个厨房（比如安装 Node.js 环境），再建一个仓库（比如装数据库）。你需要自己一块砖一块砖地垒，接水电。如果哪天你想把这个厨房搬到另一片空地，或者厨房弄得太乱你想重置，都会非常麻烦。
* **容器做法（Container）**：容器就像是一个**在工厂里已经组装好的“精装移动板房”**。这个板房里自带了所有的锅碗瓢盆（代码、运行环境、依赖库）。你只需要用吊车把它放到你的空地上，通上电，它立刻就能工作。

### 2. 容器为什么这么火？它解决了什么痛点？

在开发类似 Dehao WebPage 这样的 Web 项目，或者在服务器上部署博客时，容器有几个极其强大的特性：

* **绝不串味（隔离性）**：你可以在同一个服务器上放一个“React 前端容器”、一个“Supabase 后端容器”、一个“独立博客容器”。它们各自被锁在自己的小盒子里，互不干扰。就算其中一个崩溃了，其他的依然照常运行。
* **在哪儿跑都一样（一致性）**：很多时候代码在自己的电脑上跑得好好的，一放到服务器上就疯狂报错（因为环境不一样）。有了容器，你是在本地把整个“移动板房”打包好，原封不动地端到服务器上，彻底消灭了“在我电脑上明明没问题啊”的玄学 Bug。

### 3. 极其关键的特性：容器是“阅后即焚”的（易失性）

这一点极其重要，很多人在这里吃过亏：

**容器本身是被设计成可以随时丢弃和替换的。** 就像一个快餐盒，吃完就扔，下次再换个新的。

如果你在容器里运行了一个博客，并且把用户写的文章、上传的图片直接存在了容器的“内部”，那么**一旦这个容器被删除或者重置，里面的所有数据都会瞬间灰飞烟灭。**

**怎么解决这个问题？**
我们会用到一个叫 **“数据卷映射（Volumes）”** 的概念。相当于你在移动板房（容器）的墙上开了一个洞，接了一根管子，连到外面空地（服务器真实的硬盘）上的一个保险柜里。
这样，不管你把这个移动板房怎么拆除、怎么换新，只要新板房的管子还接着那个保险柜，你的博客内容、数据库设置就永远都在。

* **缺点**：逻辑非常复杂，容易出现两边数据对不上的“脏数据”问题。

### 二、 Supabase 数据库扩展建议：单表还是多表？

关于你问的“样品、出货资料是否同步到同一个 Supabase 表”，这是一个非常关键的数据库设计问题。

**结论：强烈建议使用“多表架构”，即不同的业务模块对应不同的 Supabase 表，而不是塞进一个大表。**

#### 1. 为什么要分表？ (架构逻辑)

虽然它们最终可能都显示在 WebPage 的不同页面，但在 Supabase (PostgreSQL) 中分开存储有以下好处：

* **数据清晰度**：PO 订单有“交期”和“单价”，样品有“打样费”和“快递号”，出货有“柜号”和“封条”。如果塞在一起，表会变得极其臃肿且难以维护。
* **性能优化**：当你查询“核销”时，数据库不需要去扫描数千条“样品”或“出货”的无关数据。
* **关联性**：它们可以通过 **`client_id` (客户唯一识别码)** 或 **`po_id`** 在逻辑上关联起来，而不是物理上挤在一起。

#### 2. 德皓 Supabase 数据库结构预想 (Schema)

| 模块 (表名) | 核心用途 | 关键字段 (Airtable 同步过来的精简版) |
| --- | --- | --- |
| **`st_orders` (PO表)** | 财务核销、交期管理 | `airtable_id`, `po_no`, `client_name`, `total_amount`, `balance` |
| **`st_payments` (收款表)** | 资金池管理 | `airtable_id`, `pay_no`, `amount`, `unallocated_amount` |

### 高级方案 1：视图驱动同步 (View-Driven Sync) —— 最优雅的过滤方式

**不要在 React 前端或 Supabase 后端写复杂的筛选代码！**
Airtable 最强大的功能之一就是它的 View（视图）。

* **怎么做**：你在 Airtable 的 `PO` 表和 `收款登记` 表里，专门新建一个视图，命名为 `API_Sync_Active`。在这个视图里，你用 Airtable 自带的 Filter 设定好条件（比如：“出货状况 != 作废” 且 “欠款余额 > 0” 且 “年份 = 2026”）。
* **API 调用**：当你在 Dehao WebPage 点击“初始化同步”时，Supabase 后端使用拥有 `data.records:read` 权限的 Token 调用 `List records` 接口 。在请求参数中，直接指定 `view: "API_Sync_Active"`。
* **为什么高级**：未来如果 Dehao 的业务规则变了（比如你想把 2025 年的尾款也加进来），你**完全不需要改任何一行 TypeScript 代码**，只需要在 Airtable 里修改那个视图的 Filter 即可，前端下一次同步会自动生效！

### 高级方案 2：增量同步 (Incremental Sync / Delta Fetch) —— 性能最优

对于日常的“手动同步”，每次都拉取全部活跃数据依然很浪费。

* **怎么做**：在 Supabase 里建一个极小的配置表 `st_sync_logs`，记录上一次成功同步的时间戳（Timestamp）。在 Airtable 表中开启 `Last Modified Time`（最后修改时间）字段。
* **运行逻辑**：点击“手动同步”时，Supabase 会向 Airtable 发送请求：**“只给我 `Last Modified Time` 大于上次同步时间的记录”**。
* **为什么高级**：每次同步的数据量从几千条锐减到可能只有几条。这让同步动作可以在一两秒内瞬间完成，极大降低了系统负载和 API 压力。

### 高级方案 3：自动化 Webhook 监听 —— 彻底解放双手

对于核心业务，“手动去点一下同步”往往存在人为遗漏的风险。最高级的形态是让系统自己保持同步。

* **怎么做**：利用 Airtable 的 Automations 或原生 Webhooks 功能（你的 Token 需要包含 `webhook:manage` 权限以查看、创建或管理 Webhook ）。
* **运行逻辑**：只要有人在 Airtable 里修改了某个订单的金额，Airtable 就会瞬间主动向你 Supabase 的 Edge Function 发送一个 HTTP POST 请求（Payload 包含了改动的数据）。Supabase 接收到后，立刻更新数据库。
* **为什么高级**：实现了真正意义上的“实时中转站”。日常根本不需要手动同步，你的 Web GUI 永远是最新的。
